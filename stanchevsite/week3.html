Week 3 Project Update

FIRST Resources

For the FIRST center we will be using a card reader as the main way of checking in and out of the space. Each team would need to be provided a card that would give them access to the space. As teams request cards they will be put into the Team table with information about the Team Number, Team Name, Main Contact (Name, phone number, etc.). There will be an activity log table that will track check-ins and check outs. This will be be keyed to the team so we can see which team the activity belongs to. There is the potential to be more specific than just the team and have a Team Member table. Each student could be given a card and as they request cards we store their Name, Phone Number, Address and link to the Team table so we know which team they belong to. This would require the activity table to link to the specific Team Members instead of the Teams. We can have a Kettering FIRST Student table that links Students Kettering ID and information with their FIRST team. This would allow us to give access to current kettering students through their IDs. All of this access could be limited based on a Permissions table with permission keys that allow use at different times, each user of the space would have a specific permission key linked to their information so we can limit the access on a per person level. This would be the requirements and potential requirements for the FIRST center.

Forums

Since each Kettering Student / FIRST Team Member has an ID after all the cards have been given out, they can use that ID to create an account on the Forums (will be a website). This allows us to link any Posts made to either a Kettering Student or a FIRST Team Member through their respective tables. The posts will show information about the person who made the post. There will need to be a Thread table that the posts are linked to, and they will be displayed once the student enters that respective thread. There will then need to be a Sub Forum table that the threads are linked under. There can be a Forum Permissions table for this as well to limit who can create Sub Forums, who can create Threads, and who can make Posts. This gives teams ways to communicate with one another about the space and projects going along in it. Since there is already a general FIRST forum already out on the internet. This will be the least priority of our 3 features.

Proposal System

The proposal system is a very interesting idea. This will once again require the Kettering Student ID or the FIRST Team Member ID to create a proposal (another website). There will be Proposal table where the information for each proposal is entered and tied to a User. There will be a Proposal Permission table that will allow specific users to enter proposals, and others to approve them. This could be limited to monetary proposals, or proposals for new tools for the space, but this proposal system can be expanded (relatively easily) to be a proposal system for anything throughout Kettering, club budgets, refund requests, ideas for the school and many other things. This makes it the most wide impacting and interesting of the three features.

Kettering Greek Life and Clubs Database

Need

Database for storage of all members of clubs including the Greek Community. Also, a way to find out

when clubs are holding meetings and if they are open to visitors.

Approach

Create a database and user interface that would allow club members to input their meeting times,

where they meet, what their club is, club name, main contact, etc. Those who are not a member of

the clubs can then sort on a certain day and time and find out what clubs they could try out. They

could also sort on a keyword topic in order to find out if there is a club out there based on their

interest. Create another section of the database that includes all members of the clubs, their positions

and contact information. Include a notes section so that if each club has different information

required, that unique information can be placed there.

The design for this database is simple. There will be three tables, two acting as entities and the third

acting as the relationship between the two. There will be a Student table and Organization table, with

the columns listed below. The third table, Participant, will have one record for every member of every

group. For example, of a student with Student ID 117 joins a group with Organization ID 343, a new

record will be added to the Participants table with those IDs, along with the position held and the year

that this person joined. This way, it would be very easy to run queries to find all of the members of a

certain group, all of the groups a particular person is a part of, and much more.

Benefits

This would improve club participation and advertisement. It would also allow people who do not

attend the Carnival of Clubs to find out when clubs are meeting and try them out. You could also do a

sort on who the presidents of all the clubs or Greek houses are and include a feature to email them all

instantly. This would help improve communication.

Competition

The Kettering calendar has some information on when and where a club meets but it is difficult to

navigate and not always accurate. As for the club participation list, that exists only on a piece of paper

or printed Excel sheet.

On Demand Radio

For this feature of our project there will need to be a screen with a set of search filters. We need to have a filter for name and a filter for date/date range. The user will be able to search for either a DJ, a date/range or both. This allows users to listen to all the shows they missed on a specific day or look for all of the shows from a specific DJ. The shows will have play buttons that will start streaming the music through the website. A neat feature to add would be a progress bar so the listener can see how far they are through the stream. Another cool feature that can be added later on in the project so that the user can make a playlist so they can queue up broadcasts to listen to them straight in a row.

Song Requests

The song requests feature of our project will be much simpler than the on demand radio feature. This one will be more of a simple form with inputs for users to input the song information they want: Title, Album, Release Date, Band, etc. A feature that would make all of this easier is a tagging system that remembers what previous users have input and allows for auto completion when the user starts typing something into the input fields.

Updating Existing Site

The rest of the existing website for WKUF could do with an update. Once/if we finish the first two features we will begin to make the existing pages look nicer and be more user friendly. This is not part of the actual project goals but would be nice to get to.
